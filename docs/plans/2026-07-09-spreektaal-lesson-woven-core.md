# Spreektaal slice 1+2 (merged): the lesson-woven register-pair core

## 0. What changed vs the program spec (owner decisions, 2026-07-09 session)

The program spec's slice 1 (grader variants only) and slice 2 (separate collection + primer)
are **restructured** by owner decision into one lesson-woven core:

1. **Informal forms are taught in the lessons themselves**, not via a separate opt-in
   collection. *nggak* lives in the lesson that teaches *tidak* — reader exposure and
   scheduling anchored to the same lesson (ADR 0006 clean).
2. **One tight core set, no tiering.** Only pairs that are (a) decades-stable
   (Sneddon-attested), (b) dominant in speech, (c) informal-standard register (safe
   everywhere) get ANY treatment. No long-tail chip-only tier. Jakarta gaul
   (*gue/lu*), particles (*dong/sih*), and post-2006 lexical slang are OUT of this
   slice (particles = program slice 3; gaul = primer-mention only, later).
3. **Toggle**: a learner can exclude spreektaal from sessions. Modeled on the shipped
   listening toggle (localStorage pref + snapshot filter) — see §5.
4. Post-2006 research (Haiku web-research pass, 2026-07-09) confirmed Sneddon's
   pair inventory is still the living core; post-2006 accretion is standalone
   social/emotional lexemes (*kepo, baper, galau, mager, gabut, lebay*), which are
   NOT register pairs and stay out of this slice.

## 1. Goal

A learner who works through the lessons meets, drills, and is never marked wrong on
the ~40–60 colloquial forms that dominate spoken Indonesian (*nggak, udah, aja, kalo,
gimana, kayak, banget, …*), each anchored to the lesson that teaches its formal twin.

Non-goal: teaching register *choice* (slice 5 / Percakapan), particles (slice 3), a
register-pair transform trainer (slice 4), churning slang (never).

## 3. Content + data model

### 3.1 The register-pairs artifact (authoring, once)

`scripts/data/register-pairs.ts` — committed, hand-reviewed artifact (the
`morphology-roots.ts` pattern). One row per pair:

```typescript
{ formal: 'tidak', informal: 'nggak', gloss_nl: 'niet',
  klasse: 'h-drop' | 'monophthong' | 'a-e-reductie' | 'frozen-verb' | 'lexical',
  source: 'Sneddon 2006 §x.y' }          // attestation note, per pair
```

- Authored by an agent from Sneddon's *Colloquial Jakartan Indonesian* (2006) +
  the systematic classes (monophthongization *pakai→pake*; h-drop *habis→abis*;
  final-syllable a→e *benar→bener*; frozen colloquial verbs *bilang/bikin/kasih/
  ngomong/ngerti/ketemu*; dominant lexical replacements *kayak/banget/duit/gimana/
  gitu/gini/emang/cuma/ntar/lagi/terus*). Full list ~100–150 rows, every row
  source-noted. Owner reviews at plausibility level (owner is not a gaul speaker —
  attestation notes are the correctness oracle).
- **The scheduled core = the deterministic intersection** of this artifact with
  taught vocabulary in the **live DB** (staging drifts — query live
  `learning_items`, per `project_staging_learning_items_drifts_from_db`). Expected
  ~40–60 pairs. A pair whose formal twin is untaught is kept in the artifact
  (future lessons may teach the twin) but produces nothing now.
- ~6 essential informal forms whose formal side is only inside taught *phrases*
  (*gimana, banget, ngerti, ngomong, bikin, kasih*): anchor each to the lesson
  containing the phrase/nearest-relative; the artifact carries an explicit
  `anchor_lesson` override for these rows. If review rejects an anchor, the row is
  deferred, not force-fitted.

### 3.2 Schema (content tables only — no learner-data surface)

Two nullable columns, each in two tables (carrier + destination), riding the
`loan_source_nl` groove:

| Column | On | Meaning |
|---|---|---|
| `register text check (register in ('informal'))` | `lesson_section_item_rows`, `learning_items` | NULL = formal/default; `'informal'` marks a spreektaal item. The CHECK is deliberately single-valued today, NOT a finished taxonomy — it widens (e.g. `'gaul'`) when Jakarta-register content arrives; widening a CHECK is a cheap additive migration |
| `register_counterpart text` | `lesson_section_item_rows`, `learning_items` | base_text of the formal twin (e.g. `'tidak'` on the *nggak* row); NULL otherwise |

- Written by the Lesson Stage from staging; forwarded to `learning_items` by the
  Capability Stage's `upsertLearningItemIdempotent` targeted-update list (same as
  `translation_nl`/`loan_source_nl` — updates apply on re-publish).
- Pre-write validator (Lesson Gate layer): `register='informal'` ⇒
  `register_counterpart` non-null ∧ counterpart resolves to a `learning_items` row
  (live), except rows with an `anchor_lesson` override where the counterpart may be
  phrase-internal — those validate counterpart-as-text against the artifact instead.
  This validator, the §4 prerequisite resolver, AND the §7 reader-union (all
  three `register_counterpart` resolution sites) MUST resolve through the
  canonical `itemSlug()` (the mint used at
  `capability-stage/projectors/vocab.ts:119`) — never a bespoke lowercase/trim —
  so slug edge cases (*kayak*, hyphenated forms) can't diverge (data-architect
  r2 + r3 addendum).
- ⚠️ Like `translation_nl`/`loan_source_nl`, both columns are **rewritten from
  staging on every publish of the anchor lesson** — a direct DB edit to them is
  silently clobbered next publish. Edit staging, not the DB.
- `translation_nl` on informal items stays a **clean, gradeable gloss** (`'niet'`,
  not `'niet (spreektaal)'`) — the 2026-07-05 gloss-pass rule; the register mark is
  the typed column, never prose in the gloss.

### 3.3 Staging weave (the backfill)

For each core pair: append one item entry to
`scripts/data/staging/lesson-N/learning-items.ts` of the anchor lesson (the formal
twin's introducing lesson), with `register`/`register_counterpart` fields. Then
per-lesson: Stage A + Stage B publish (additive, idempotent — the normal loop per
`feedback_publish_loop_per_lesson`), and re-fetch that lesson's `content.json`.
~15–20 lessons touched, concentrated in L1–L10. Agent-driveable; the per-lesson
chapters-content-parity test is the guard.

## 4. Scheduling

- **Informal items are receptive-only (staff-engineer correction, r1).** They generate
  the recognise caps only — #1 `recognise_meaning_from_text_cap` and #3′
  `recognise_meaning_from_audio_cap` — NOT #2 (`recognise_form_from_meaning_cap`)
  and NOT #6 (`produce_form_from_meaning_cap`). Why: under §7's bidirectional
  acceptance, an informal #6 card is a near-duplicate of the formal item's #6
  (identical NL prompt, identical accepted set) — review load with no new teaching,
  the exact redundancy the four-card-ladder work (ADR 0027) just removed. The
  skill this slice sells is *understanding the reply* — receptive. Producing
  informal register is slice 4's trainer. Mechanism: one generation rule in the
  capability stage keyed on `register='informal'` (per-source-kind cap-set
  selection already exists for podcast items; this is the per-item analogue).
  #1 graduates via #3′ under the existing `#1 ← (#3′ ∨ #6)` OR-rule — no
  graduation change needed.
- **Audio gate:** #3′ depends on TTS quality for colloquial forms (program spec
  open Q3 — does Chirp3-HD render *nggak* naturally?). One-time manual spot-check
  of ~5 forms before step 4; if TTS fails, informal items ship text-only #1 (a
  non-graduating scaffold, accepted for ~50 deliberately-cheap cards until
  slice 4 provides typed successors).
- **Distractor guard (required regardless of cap choice):** the formal item's #2
  MCQ (`choose_form_ex` — prompt *'niet'*, choose the Indonesian) can now draw the
  informal twin from the runtime distractor pool — two correct choices. The guard
  lands in the `choose_form_ex` assembly (`src/lib/exercise-content/byType/`
  `cuedRecall.ts` + `byKind/item.ts` — architect r2 seam correction: NOT
  `recognitionMcq.ts`, whose `choose_meaning_ex` same-meaning dedup at
  `recognitionMcq.ts:106` already drops the twin): exclude any pool item that is
  the answer item's `register_counterpart` or names it as counterpart. Curated
  distractor sets get the same rule as a pipeline validator check.
- **Known graduation limitation (architect r2 W1, accepted + enumerated in §8):**
  for listening-disabled learners the listening filter strips #3′ and there is no
  #6, so an informal #1 never graduates — a lifelong cued MCQ, which ADR 0027
  generally forbids. Accepted deliberately for this bounded set (~40–60
  transparent register cards, only for the listening-opt-out population) rather
  than going #3′-only, which would give listening-disabled learners no spreektaal
  cards at all; slice 4's register trainer adds the typed successor that lets
  these #1s graduate later.
- Informal items carry the anchor lesson's `lesson_id`; lesson activation makes
  them introducible exactly like the rest of the lesson (gate at
  `pedagogy.ts:398-405`).
- **Formal-first ordering:** the Capability Stage adds the formal twin's phase-1 cap
  canonical key to each informal cap's `prerequisiteKeys` (resolved via
  `register_counterpart` at generation time — mirror of the morphology root-vocab
  prerequisite). Effect: *nggak* becomes introducible only after *tidak* has ≥1
  successful review (`satisfiedKeys` gate). Phrase-anchored rows (no item twin)
  get no extra prerequisite.
- Load: ~40–60 extra items ≈ 1.5–2 per lesson, and they are cheap cards
  (transparent link to a known word). No budget changes.

## 5. The toggle (spreektaal aan/uit)

Modeled 1:1 on the shipped listening toggle:

- `src/lib/spreektaalPreferences.ts` — localStorage-backed, **default true**
  (spreektaal included). Per-device, like `listeningPreferences.ts` — accepted
  trade-off, same as listening.
- `src/lib/session-builder/spreektaalFilter.ts` — when disabled, filters the loaded
  snapshot exactly like `excludeListeningCapabilities`: removes matching rows from
  `schedulerRows` (kills due + practice passes) and `plannerInput.readyCapabilities`
  (kills new introductions). Match key: capability `sourceRef` ∈ the informal-item
  ref set.
- The informal-item ref set: derived client-side from one small query
  (`learning_items` where `register='informal'` → source_refs, ~60 rows, cacheable
  per session build). The parity-locked RPC is untouched.
- Prerequisite safety (the property `listeningFilter.ts:19-24` documents): informal
  caps are never a prerequisite of anything (formal is prereq of informal, never the
  reverse), so filtered-out state rows are inert in `buildUnlockedSourceRefs`.
- Toggle OFF semantics, stated for the spec record: stops introductions and mutes
  reviews in sessions; FSRS state keeps aging; surfaces computing "due" outside the
  builder (dashboard badge, Voortgang forecast) will still count muted cards.
  v1 accepts this (the listening toggle has the same property); enumerated in §8
  as a known limitation, not silently.
- Profile UI: one switch next to the listening toggle, copy
  "Spreektaal (informele woorden) oefenen".

## 6. Display

- **Reader:** informal items appear in the anchor lesson's vocab section like any
  item (they're in `content.json` after re-fetch). `scripts/fetch-lesson-content.ts`
  threads `register`/`register_counterpart` into the snapshot; the vocab-grid
  rendering in each touched bespoke page shows a small `spreektaal` tag on informal
  rows (one shared CSS-module pattern; the per-lesson page edit rides the same
  agent pass as the snapshot re-fetch). Chapter parity tests updated per lesson.
- **Exercise surfaces:** no new mechanism — the item renders like any vocab card.
  The card's item metadata shows the `spreektaal` tag where POS/level chips already
  render, so a learner always knows they're looking at an informal form.
- The reverse chip on the FORMAL word ("spreektaal: *nggak*") is **cut from v1**
  (omission test: nothing breaks — the informal form is already visible two rows
  down in the same vocab list). Candidate polish later.

## 7. Grader acceptance

For every **core pair** (not the full artifact — owner scoped this to the core):

- `item_answer_variants` row on the FORMAL item: `variant_text = informal form`,
  `language = 'id'`, `variant_type = 'informal'` (exists in the CHECK),
  `is_accepted = true` — typing *nggak* where *tidak* is expected is correct.
  This implements the program's cross-slice contract (open Q4: vocabulary graders
  accept both registers; register-appropriateness is only tested where register IS
  the exercise — slice 4/5).
- NO mirror ID-side row on the informal item: informal items are receptive-only
  (§4), so no card ever asks for a typed Indonesian answer against the informal
  item — the row would grade nothing (omission test). Revisit with slice 4.
- NL-side: **no copy** (data-architect r2 CRITICAL — a copied variant set is a
  second, unsynced instance of the same fact that drifts the moment the formal
  item's variants are corrected via flag→review). Instead the runtime reader
  unions: `fetchAnswerVariants` (`src/lib/exercise-content/byKind/item.ts:140-145`)
  is extended so that for a `register='informal'` item it resolves
  `register_counterpart` to the formal item and includes that id in the existing
  `.in()` batch — the #3′ ear-typed NL recall then grades against the formal
  twin's living variant set. One read-path union, zero seed-then-resync mechanism.
  For the ~6 phrase-anchored rows (§3.1) whose counterpart is text-only and does
  NOT resolve to an item, the union no-ops — the informal item grades against its
  own (possibly empty) variant set plus its clean `translation_nl` (data-architect
  r3 addendum). The union behavior gets a unit test in
  `src/lib/exercise-content/__tests__/` (runtime behavior — the right validator
  layer is a test, not a live-DB health check).
- Seeded by extending the shipped `scripts/enrich-answer-variants.ts` **apply**
  path with a deterministic reader of `register-pairs.ts` (no LLM — closed list;
  same `ON CONFLICT DO NOTHING` idempotent insert). This seed now writes ONLY the
  formal-item `informal` rows above.

Grammar produce exercises (`acceptable_answers`) are NOT handled here — that is the
companion spec `2026-07-09-g4-produce-grader-fix.md`, which consumes the same
`register-pairs.ts` artifact.

### homelab-configs changes
- [ ] PostgREST: N/A (existing `indonesian` schema).
- [ ] Kong/GoTrue/Storage: N/A (no new origins, auth, buckets).

### Health check additions
- `check-supabase-deep.ts`:
  1. every `learning_items.register='informal'` row has non-null
     `register_counterpart`;
  2. every core pair in the committed artifact with a live formal twin has its
     formal-item `item_answer_variants` row (§7 seed completeness);
  3. informal rows' counterpart resolves to a live item OR the artifact marks it
     phrase-anchored;
  4. **cardinality vs the committed intersection report** (data-architect r2
     CRITICAL-1a): `count(learning_items where register='informal')` matches the
     step-2 intersection artifact — catches the whole six-file carrier chain
     silently no-oping (all other checks pass vacuously on zero rows);
  5. **no forbidden caps for informal items** (CRITICAL-1b): zero
     `learning_capabilities` rows with `capability_type IN
     ('recognise_form_from_meaning_cap','produce_form_from_meaning_cap')` whose
     `source_ref` resolves to a `register='informal'` item — catches the
     generation carve-out not landing (silent #2/#6 near-duplicates);
  6. **prereq keys resolve** (architect r2 W4): every informal cap's
     `prerequisiteKeys` entries resolve to an existing published capability —
     catches slug-mint drift that would leave *nggak* `missing_prerequisite`
     forever (`pedagogy.ts:329`).

### Known limitations (accepted v1)
- Toggle is per-device (localStorage), and muted cards still age in FSRS and count
  in due badges outside the builder (listening-toggle parity).
- No `suspended`-state flip; no register facet in Voortgang (coverage-first, per
  program spec open Q5).
- Listening-disabled learners' informal #1 cards never graduate (no #3′, no #6) —
  a bounded, deliberate ADR-0027 exception (~40–60 cheap cards, opt-out population
  only) until slice 4 adds typed successors. See §4.

## 9. Build order (each step commits green)

Gates: any change to `scripts/migration.sql` runs `make migrate-idempotent-check`
before merge; `make migrate` applies (chains deep health checks); `make pre-deploy`
before merging each PR (architect r2 W5).

1. Migration (4 columns) + validators + health checks (three-layer gate, one PR).
2. `register-pairs.ts` artifact authored + reviewed; intersection report against
   live DB committed alongside.
3. Pipeline carrier: lesson-stage projection + capability-stage forward + prereq
   wiring + tests.
4. Staging weave + per-lesson publish + snapshot re-fetch + page tag (agent pass,
   ~15–20 lessons, parity tests green per lesson).
5. Variant seed (extend enrich-answer-variants apply) + health check green.
6. Toggle: preferences + filter + Profile switch + tests.

Steps 5 and 6 are independent of 4 and can land in parallel after 1–3.
