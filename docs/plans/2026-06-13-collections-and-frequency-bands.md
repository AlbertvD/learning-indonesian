# Collections: selectable word-lists (frequency bands + thematic packs)

> **Revision note (round 1 → 2):** corrects the gate seam cite (`pedagogy.ts:gateCandidates`, not a non-existent `eligibility.ts`), rewrites the pipeline section to the real Stage-A→`lesson_section_item_rows`→item-projector path (drops the false GT10 claim), specifies gate-clause ordering vs. the home-lesson, adds the `lessons.is_hidden` mechanism (C1), the `source_type` value (M1), the bidirectional projection-drift gate (M3), and full RLS/grant DDL. See the two round-1 reviews.

## 1. Goal

Let a learner **select extra word-lists to study** alongside the 16 lessons — starting with **frequency bands** ("Top-100/300/500/1000/2000 most common words"), and later **thematic packs** ("Holiday", "Food"). One mechanism serves both. It is also the **monetization seam**: collections are the future SKUs (entitlements gate their activation — out of scope here, but the `set_collection_activation` RPC must stay the clean chokepoint). See `docs/roadmap.md`.

**Immediate driver:** the Top-1000 band (the declared first content target). The feature ships against existing vocabulary first; the ~365 gap words fill it over time.

## 2. Operating-context note

Build-stage rules still apply (`CLAUDE.md` Operating Context): disposable data on the homelab, truncate-and-rebuild freely, no live-system safety machinery. Design the schema cleanly (it outlives the disposable phase and becomes the paywall boundary), but no backfill/coexistence choreography.

## 3. The core insight (why this is small)

- **FSRS is source-agnostic** (ADR 0003 / ADR 0006: *"lesson_id is metadata for the planner, not the scheduler"*). A collection word schedules identically to a lesson word. **No scheduler change; no `canonical_key`/`projection_version` bump.**
- **Word identity is already global** — `learning_items.UNIQUE(normalized_text)` (`scripts/migration.sql:133`). A word in lesson 2 *and* Top-100 is **one** item, **one** capability set, **one** FSRS state. Overlap cannot be duplicated by construction; existing progress is shared.
- So this is a **selection + activation + grouping** feature, not a scheduling one. Work: a data model, a thin `lib/collections/` module, one reworked clause in the new-introduction gate, and a vocab-only ingestion path for non-lesson words.

## 4. Data model

### 4.1 New column on `learning_items`
```sql
alter table indonesian.learning_items
  add column if not exists frequency_rank int;   -- canonical "how common" signal; NULL = unranked
```
Item-level (frequency is a property of the word, not its capabilities — avoids the per-capability duplication `lesson_id` carries). It is the **canonical input** the frequency-band membership is projected from (§4.4).
*Out of scope:* using `frequency_rank` for session ordering ("teach commoner words first") is a separate planner change with no design here — explicitly NOT wired into `prioritizeCandidates`.

### 4.2 New value on `learning_items.source_type` (M1)
Gap words belong to no coursebook lesson. Extend the existing CHECK (`migration.sql:126`, currently `'lesson','podcast','flashcard','manual'`) to add `'collection'`:
```sql
-- in migration.sql, replace the CHECK:
source_type text NOT NULL CHECK (source_type IN ('lesson','podcast','flashcard','manual','collection'))
```
Gap words carry `source_type='collection'`.

### 4.3 New tables
```sql
create table if not exists indonesian.collections (
  id           uuid primary key default gen_random_uuid(),
  slug         text not null unique,                 -- 'top-1000', 'holiday'
  name         text not null,
  kind         text not null check (kind in ('frequency','theme')),
  rank_cutoff  int,                                  -- set iff kind='frequency'; INCLUSIVE (rank <= cutoff)
  is_published boolean not null default true,
  created_at   timestamptz not null default now(),
  check (kind <> 'frequency' or rank_cutoff is not null)
);
comment on column indonesian.collections.rank_cutoff is
  'Frequency-band threshold, INCLUSIVE: an item is a member iff learning_items.frequency_rank <= rank_cutoff. NULL for kind=theme.';

-- ONE membership table for ALL collections.
--   frequency: rows GENERATED by the pipeline from frequency_rank <= rank_cutoff (a projection of 4.1)
--   theme:     rows AUTHORED directly
create table if not exists indonesian.collection_items (
  collection_id    uuid not null references indonesian.collections(id) on delete cascade,
  learning_item_id uuid not null references indonesian.learning_items(id) on delete cascade,
  primary key (collection_id, learning_item_id)
);
create index if not exists collection_items_item_idx
  on indonesian.collection_items(learning_item_id);

create table if not exists indonesian.learner_collection_activation (
  user_id       uuid not null references auth.users(id) on delete cascade,
  collection_id uuid not null references indonesian.collections(id) on delete cascade,
  activated_at  timestamptz not null default now(),
  primary key (user_id, collection_id)
);
create index if not exists learner_collection_activation_user_idx
  on indonesian.learner_collection_activation(user_id);
```

**RLS + grants (idempotent, per the Migration source-of-truth rule):**
```sql
-- content tables: world-readable to authenticated, pipeline (service_role) writes
grant select on indonesian.collections, indonesian.collection_items to authenticated;
alter table indonesian.collections enable row level security;
alter table indonesian.collection_items enable row level security;
drop policy if exists "collections read" on indonesian.collections;
create policy "collections read" on indonesian.collections for select to authenticated using (true);
drop policy if exists "collection_items read" on indonesian.collection_items;
create policy "collection_items read" on indonesian.collection_items for select to authenticated using (true);
grant all on indonesian.collections, indonesian.collection_items to service_role;

-- per-learner selection: owner-read, writes only via RPC
alter table indonesian.learner_collection_activation enable row level security;
drop policy if exists "collection activation owner read" on indonesian.learner_collection_activation;
create policy "collection activation owner read"
  on indonesian.learner_collection_activation for select to authenticated using (user_id = auth.uid());
grant select on indonesian.learner_collection_activation to authenticated;
revoke insert, update, delete on indonesian.learner_collection_activation from authenticated;
grant all on indonesian.learner_collection_activation to service_role;
```

**Selection write path** — `set_collection_activation`, mirroring `set_lesson_activation` (`migration.sql:1877`):
```sql
create or replace function indonesian.set_collection_activation(
  p_user_id uuid, p_collection_id uuid, p_activated boolean
) returns void language plpgsql security definer
set search_path = indonesian, public as $$
begin
  if p_user_id is null or p_collection_id is null or p_activated is null then
    raise exception 'set_collection_activation requires p_user_id, p_collection_id, p_activated';
  end if;
  if coalesce(auth.role(),'') <> 'service_role' and auth.uid() is distinct from p_user_id then
    raise exception 'not authorized';
  end if;
  if p_activated then
    insert into indonesian.learner_collection_activation(user_id, collection_id)
    values (p_user_id, p_collection_id) on conflict do nothing;
  else
    delete from indonesian.learner_collection_activation
    where user_id = p_user_id and collection_id = p_collection_id;
  end if;
end $$;
revoke all on function indonesian.set_collection_activation(uuid, uuid, boolean) from public;
grant execute on function indonesian.set_collection_activation(uuid, uuid, boolean) to authenticated, service_role;
```
(This RPC is the future entitlement chokepoint — keep it clean. The `revoke … from public` before `grant execute` mirrors `set_lesson_activation`, `migration.sql:1911`.)

**Why one membership table** (not a query-time `rank ≤ cutoff` rule): the codebase's content model is "canonical input → projected rows." Materialising frequency membership gives the gate **one uniform join** instead of two membership mechanisms + a per-kind branch. ~4k learner-independent rows. Both reviewers endorsed this from their lenses. `collections.kind` (a CHECK'd column) is the discriminator if a reader needs to restrict.

### 4.4 Gap-word home + the `is_hidden` mechanism (C1, OQ#1)
`learning_capabilities.lesson_id` is 1:1 and `NOT NULL` for non-podcast caps (ADR 0006 CHECK). Frequency words not in any lesson need a home. **Decision (both reviewers concur): a single hidden "Common Words" `lessons` row** — keeps the ADR 0006 invariant intact (vs. a CHECK carve-out, which would make null-lesson caps bypass the activation gate entirely and expose all gap words to every learner — see OQ#1). But the hidden lesson needs three supporting changes the readers of `lessons` require:

```sql
alter table indonesian.lessons
  add column if not exists is_hidden boolean not null default false;
```
1. **`get_lessons_overview` filter** (`migration.sql:~1976`) — add `where not l.is_hidden` so the Common Words row never renders as a lesson card. (It currently returns every `lessons` row unconditionally.) Implement as a verbatim idempotent `drop function if exists …; create or replace function …` with `where not l.is_hidden` in the body — the `RETURNS TABLE` shape is unchanged.
2. **Lessons UI** — consumes `get_lessons_overview`; honour the filter (no separate change if the RPC filters).
3. **Non-collision + no auto-activation** — the sign-in backfill auto-activates `order_index in (1,2,3)` with **no module filter** (`migration.sql:1937`). **Concrete seed: a dedicated `common-words` module with the Common Words lesson at `order_index = 999`** — any `order_index > 3` in any module is safe from the backfill, and a dedicated module avoids the `UNIQUE(module_id, order_index)` collision (`migration.sql:91`).

The Common Words lesson doubles as the **ingestion vehicle** (§6). Its caps carry its `lesson_id`; since no learner activates it, they surface **only** via the collection clause (§5) — which is the intended gating.

## 5. Runtime: the eligibility gate (`lib/session-builder/`)

**Live seam (corrected):** the new-introduction gate is `src/lib/session-builder/pedagogy.ts` → `gateCandidates` (the `lesson_not_activated` suppression at `pedagogy.ts:373-376`). There is **no `eligibility.ts`** today (target-arch *plans* that fold; when it happens it carries this clause).

**The change — and clause ordering is load-bearing.** Today: `if (lessonId != null && !activatedLessons.has(lessonId)) suppress('lesson_not_activated'); continue;`. Because gap-word caps carry the Common-Words `lessonId` that **no learner activates**, that clause would suppress every gap word. Rework so a cap is suppressed **only if it passes neither gate**:

> suppress `not_activated` iff `!(lessonId ∈ activatedLessons) AND !(itemId ∈ anyActivatedCollection)`.

i.e. collection membership is an **OR** that rescues collection-member caps whose (home or real) lesson isn't activated. This must be expressed as concrete clause placement in `gateCandidates`, with the named test in §9. (This is the ordering-bug class that cost a session-builder day — sibling-bury-before-allocate; treat it as such.)

**New `lib/collections/` module (OQ#2 — resolved: new module, do NOT fold into `lib/lessons/`).** Grounding correction: `lib/lessons/`'s barrel re-exports **~22 runtime symbols** (`src/lib/lessons/index.ts`), already over the ~10 ceiling in `docs/target-architecture.md` §"Depth and width rules" rule 3 — folding a second noun violates §3 "one job per module." Collections is its own noun, sibling to lessons. It exposes `listActivatedCollections(userId)` + membership resolution, paralleling `lib/lessons/listActivatedLessons`. **It must NOT import `lib/session-builder/`, `lib/scheduling/`, or `lib/analytics/`** (Rule 7 / no back-edges) — same posture as `lib/lessons/`; session-builder consumes collections, not vice-versa.

**Adapter:** the session-builder adapter adds reads of `learner_collection_activation` + `collection_items` to the snapshot it already assembles (alongside `learner_lesson_activation`).

**Capability→item resolution (OQ#3 — resolved: no FK yet).** `learning_capabilities` has no `learning_item_id` column; item caps link via `source_ref = 'learning_items/<normalized_text>'` (`itemSlug` is the sole slug derivation; `vocab.ts:41`). The adapter resolves membership by: filter `source_kind='item'` → strip the `learning_items/` prefix → join `learning_items ON normalized_text` (UNIQUE, `migration.sql:133`) → join `collection_items ON learning_item_id`. The UNIQUE constraint gives the same guarantee as a FK, so **a denormalized `learning_item_id` FK is deferred** (future optimization, not needed now). This resolution path is documented here so the adapter implementation is unambiguous.

**Hot-path note:** resolution is a single indexed join in the once-per-build snapshot load (not the in-memory gating loop). `collection_items` is learner-independent → join cost constant in learner count. Negligible to ~10k learners; the ADR 0015 server-side RPC is the lever beyond that.

## 6. Content: the vocab-only "Common Words" ingestion path (corrected)

**Corrected pipeline model** (round-1 architect finding). The capability-stage does **not** read a `{base_text, translation_nl}` blob — it reads `lesson_section_item_rows` from the DB (`scripts/lib/pipeline/capability-stage/.../loadFromDb.ts`), which **Stage A (lesson-stage) writes from a `vocabulary` section**. So Common Words words reach the pipeline as a **Stage-A `vocabulary` section** in a staging unit → `lesson_section_item_rows` → the existing capability-stage **item projector** (`projectors/vocab.ts`) emits the capability suite + distractors + an **anchor `item_context`** per item (which satisfies the "every published item has ≥1 context" check, `verify/seedIntegrity.ts`). **No dedicated pipeline path is needed.**

- Staging human input per gap word: `{ base_text (id), translation_nl }`. Pipeline enriches `translation_en` (existing EN enricher), POS, level, `normalized_text`, `frequency_rank`. **NL is author-reviewed (answer key); EN enriched.** No dialogue, no grammar, **no cloze** (vocab/item cloze is won't-build; `memory/project_cloze_surface_map`).
- **Item type:** gap words must be `word`/`phrase` item_types — per **ADR 0014**, only those get the capability suite. (They will be; cite for the implementer.)
- **The gate claim, corrected:** the lesson-stage/capability-stage gates are **per-section-type** and fire only for sections *present* (`validators/sectionType.ts`); coverage gates (grammar CS18 `patternCoverage.ts`, dialogue-cloze CS22 `dialogueClozeCoverage.ts`) only map over patterns/lines actually written → **a vocab-only unit trips none of them.** The earlier "GT10 fails a grammar section with no categories[]" claim was **false** (GT10 is `validateDisplayContentShape`). So the feared "gate-acceptance change" is largely **not required** — verification task: confirm a vocab-only unit publishes clean through both stages; only add a gate change if a *specific* gate is shown to reject it.

**Corpus source (out of this spec):** PBWL (CEFR leveling + word list + affix tags) pending the author's commercial-permission reply (requested 2026-06-13); fallback OpenSubtitles + own leveling. Corpus choice determines *which words fill the model*, not the model. (`memory/project_monetization_direction`.)

## 7. Overlap / dedup — the normalization-key contract

Deck-build is **resolve-or-create on `normalized_text`**, never blind insert:
1. normalize the incoming word → look up `learning_items.normalized_text`;
2. **found** → add a `collection_items` row pointing at the existing item (no new item; FSRS progress shared);
3. **not found** → create the item (`source_type='collection'`, Common Words home), then add membership.

`UNIQUE(normalized_text)` is the hard backstop. **Correctness risk:** the normalization key MUST match the lesson pipeline's exactly (root vs. surface, *baca* vs *membaca*) or a near-duplicate is created. The deck-build MUST reuse the lesson pipeline's normalization/lemmatization (`itemSlug`). Single biggest correctness item.

## 8. Three-layer invariant gates (per `memory/project_three_layer_invariant_gates`)
1. **Shared helper + unit tests:** the frequency-projection function (`membership(collection) == { items | frequency_rank ≤ rank_cutoff }`) and resolve-or-create.
2. **Pipeline pre-write validator:** no orphan `collection_items` (every `learning_item_id` resolves to a published item); for each frequency collection, the materialised rows equal `rank ≤ cutoff` **in both directions** (M3): (a) every row has `frequency_rank ≤ rank_cutoff`, **and (b) every item with `frequency_rank ≤ rank_cutoff` is present** (catches stale projection after a `frequency_rank` update).
3. **Live-DB health check (`check-supabase-deep`):** RLS + grants on the three tables; both projection directions from gate 2; every `learning_capabilities.source_ref` with the `learning_items/` prefix resolves to a real `normalized_text` (validates §5's resolution path).

## 9. Test scenarios (concrete)
- **Gate OR:** a learner who activated *Top-1000* sees gap-word caps in `buildSession` (whose home lesson is the un-activated Common Words lesson). *(guards the §5 clause-ordering bug.)*
- **Deactivation:** deactivating *Top-1000* removes its gap-word caps from the next `buildSession` but leaves **lesson-overlap** words (shared FSRS state) untouched.
- **Resolve-or-create:** building a collection that includes an existing lesson word returns the existing `learning_items` row (no duplicate; `UNIQUE(normalized_text)` holds).
- **Hidden lesson:** the Common Words lesson does NOT appear in `get_lessons_overview` output and is NOT auto-activated on sign-in. *(Test under `set local role authenticated` + `request.jwt.claims` — `get_lessons_overview` is SECURITY INVOKER over RLS-protected tables, so the role context is load-bearing.)*
- **Frequency projection:** updating an item's `frequency_rank` past a cutoff and re-materialising adds/removes exactly the right `collection_items` rows (both directions).
