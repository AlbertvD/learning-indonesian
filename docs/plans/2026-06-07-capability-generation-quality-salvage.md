# Capability-generation quality salvage (in place — no rebuild)

## Decision this spec implements
**Do not continue the cap-v2 capability-stage rebuild/strangler for quality reasons.** A code-grounded A/B review (2026-06-07) showed the perceived quality regression is NOT structural and NOT "LLM→deterministic" wholesale — it is the **dropped reviewer/critique pass** plus a few per-generator flaws, all fixable in place. See `memory/project_capability_quality_salvage_not_rebuild.md`. cap-v2 Slice 1 (vocab module) stays as merged; this spec stops further slicing and fixes generation quality directly.

## Evidence (A/B: in-stage generators vs the real old `cloze-creator`/`grammar-exercise-creator`/`vocab-exercise-creator` → `linguist-reviewer`, same model claude-sonnet-4-6, lesson 7)
1. **Dialogue cloze — real regression.** Carrier = the whole dialogue line (`generateClozeContexts.ts`), and the textbook's "lines" are often multi-sentence turns/paragraphs → live corpus has 13/76 clozes over the 16-token ceiling (worst: a 60-token 8-sentence paragraph, L7 `tiba`), 53/76 multi-sentence. The OLD `cloze-creator` agent ALSO uses whole-line carriers + no ceiling → reproduced the same paragraph. The differentiator was the **reviewer**, which caught every defect and whose fix converged on single-sentence extraction.
2. **Grammar — basically fine.** Short clean source sentences in both arms; verbose explanations in *both* (in-stage med ~190 chars, agent med 174) → a shared prompt choice, not a regression. Optional conciseness cap only.
3. **Distractors — minor, real regression on the LIVE pointer path.** Re-grounded after round-1 review (see Review history R1/C1). `vocabulary/selectDistractors.ts` already excludes the answer, its morphological variants (shared `stripAffixes` root, `selectDistractors.ts:44-53`), and near-synonyms (cosine ≥ `DEFAULT_SYNONYM_THRESHOLD` 0.85, `:148-149`) — so "answer-leak" and "near-synonym" are **already handled**; my round-0 wording was imprecise. The defect that **survives** is **cross-distractor rendered-text duplication**: distractors are stored as item-id pointers (`distractors(capability_id, item_id)`) but **rendered** as the item's `translation_nl` (meaning MCQ) or `base_text` (form MCQ), and two *distinct* item_ids can share a rendered string with **no cross-distractor dedup**. Verified live: **5/43** L7 `text_recognition` caps show a duplicate rendered gloss (`rood|rood`, `gestreken|gestreken`, `munteenheid|munteenheid`×2, `maar/echter|maar/echter`), all with distinct item_ids. Secondary data-hygiene defect: a stray item `disetrika*` (distinct from `disetrika`) the root-strip misses, and a de-harvest-leak full-sentence item — see F1b.

**Unifying finding:** the cheapest mechanism that recovers quality is (a) deterministic validators/prompt rules per generator + (b) a *scoped* reviewer pass only where judgment is genuinely required (cloze answer-uniqueness). Per `feedback_minimum_mechanism`: prefer the validator over a reviewer wherever a validator suffices.

## The fix (ordered by leverage / cheapest-first)

### F1 — Distractors: cross-distractor rendered-text dedup (the verified net-new gap)
**Net-new only — do NOT re-add what already exists.** `selectDistractors.ts` already excludes answer/variant/synonym by identity/root/embedding; the dead text-array path (`generateItemDistractors.ts`, imported only as a *type* at `runner.ts:62`; never called post-cutover) and its validator CS16 (`validators/itemDistractors.ts`, only runs when `gate.ts:212 input.distractorSets` is populated — the live item path never populates it) are vestigial. The live coverage validator is CS15 (`vocabulary/validateCoverage.ts`).
- **Enforcement seam — chosen deliberately (C8):** dedup at **seed time** in `selectDistractors.ts` + the **F1c health check** as the backstop for already-written rows. **No runtime render-time dedup** — the reader (`byKind/item.ts` → builders) just resolves `item_id → translation_nl/base_text` and renders; data-architect confirmed no builder change is needed. Single seam, no parallel enforcement.
- **The dedup helper** in `selectDistractors.ts`: after selection, dedup the chosen set by what will render — `translation_nl` for `selectMeaningDistractors`, `base_text` for `selectFormDistractors` — and also exclude any candidate whose **rendered string equals the answer's rendered string** (not just its embedding/forms). Under-supply falls back to `min(k, |eligible-after-dedup|)` (existing convention). Shared helper + unit test (a set with two same-gloss distinct item_ids → one survives).
- **Pre-write validator (C7) — DROPPED as vacuous (build-time Minimum-Mechanism reduction).** Verified at build: `seedDistractors` is the **single deterministic writer** to `distractors`, and it calls the now-deduped selectors via `planDistractorWrites` (`planDistractors.ts:94-105`). A pre-write validator re-checking the in-memory selection would re-assert exactly what `dedupeByRendered` guarantees on the same path — the vacuous-gate anti-pattern (CLAUDE.md Minimum Mechanism). The architect's approved C8 seam is **"seed-time dedup + F1c backstop"** (the C7 "sibling validator" was conditional on adding one — *"if* you add a validator, don't extend the count-only CS15"). Omission test: nothing the dedup unit-tests + F1c don't already cover. So the enforcement is **two layers, not three**: layer-1 = `dedupeByRendered` + its unit tests (the guarantee); layer-3 = F1c (independent — reads *written* rows, catches stale rows + any future writer that bypasses the selector). CS15 stays count-only, untouched.
- **Retire the dead path surgically (C6 — corrected enumeration):** `DistractorInputItem` is defined in `generateItemDistractors.ts:58` and imported by **both** `runner.ts:62` and `loadFromDb.ts:23` — it is the live return type of `fetchDistractorPool` (`loadFromDb.ts:260`) feeding the grammar/pattern path (`runner.ts:184,400-404`). So **do NOT delete the file wholesale** (my round-2 enumeration was wrong). Instead: (1) relocate `DistractorInputItem` **and `GenerateFn`** (both imported from the doomed file at `runner.ts:62`; `DistractorInputItem` also at `loadFromDb.ts:23`) to a live module (`loadFromDb.ts` or a shared types file) and repoint `runner.ts:62/100` + `loadFromDb.ts:23`; (2) THEN delete the dead runtime `generateItemDistractors()` function (`:357`) + prompt/parse/batch helpers + its test, and the unreachable CS16 `validateItemDistractors` (`validators/itemDistractors.ts:73`, only runs when `gate.ts:212 input.distractorSets` is populated — never on the live path). **Complete consumer enumeration for the deletion (R3/C6 — all verified, must be removed in the same change or the build/suite breaks):** (a) `gate.ts:62` import of `validateItemDistractors` + `ValidateItemDistractorsInput`; (b) `gate.ts:97` the `distractorSets?: ValidateItemDistractorsInput` field on the gate input; (c) `gate.ts:209-214` the CS16 comment + call block; (d) `__tests__/validators/itemDistractors.test.ts` (the CS16 *validator* test, whole file); (e) `__tests__/enforcement/noDiskReads.test.ts:105` (`generateItemDistractors.ts`) **and** `:107` (`validators/itemDistractors.ts`) — both assert the file EXISTS, so they go red on deletion; (f) `__tests__/generateItemDistractors.test.ts` (the *generator-function* test — distinct from (d); imports `buildPrompt`/`parseResponse`/`generateItemDistractors`/`ItemDistractorSet`, whole file). **Completeness check (do not enumerate further by hand): after the deletion + relocation, `grep -rl 'generateItemDistractors' scripts/lib/pipeline/capability-stage/` must return only intentional residue, then clean the comment-only refs (`generateClozeContexts.ts`, `generateGrammarExercises.ts`, `runner.ts`, `loadFromDb.ts:964`, `selectDistractors.ts`) and the operator-facing log message at `validators/itemCoverage.ts:59` ("check generateItemDistractors logs" → no longer valid), and run the full suite green.** This removes the parallel-validator drift `project_three_layer_invariant_gates` warns against without breaking the live type.
- **Omission test:** without dedup, meaning-MCQ items render two identical options (`rood|rood`) → an unfair item where the "wrong" duplicate is indistinguishable. A deterministic dedup (NOT a reviewer) is the cheapest enforcement.

### F2 — Cloze: single-sentence carrier extraction (requires an invariant rewrite)
In `generateClozeContexts.ts`:
- Carrier = **the single self-contained sentence containing the blank**, extracted from the dialogue line (proven by the A/B to collapse every long paragraph: L13 60t→9t). Keep the 16-token ceiling as a backstop on the extracted sentence; add a token **FLOOR** to reject over-trimmed/under-context carriers (the A/B's 3-token `"Kalau ___ bagaimana?"`).
- **C3 — IMPLEMENTED as narrow-after-faithful-whole-line (a safer refinement of the approved "retire reconstruction + extract").** Rather than retiring the exact-reconstruction guard (which C3 noted would drop every extracted cloze), the build **keeps** it: the LLM still returns the whole line with one blank, `sanitizeGeneratedCloze` still checks it reconstructs `line.text` exactly (the LLM-faithfulness guard, unchanged), and a new pure `narrowClozeCarrier` then narrows the carrier to the blank's sentence **only when** the line exceeds `DIALOGUE_CLOZE_MAX_TOKENS`. The narrowed carrier is a sentence-substring of the faithful whole line ⇒ a **contiguous span** of `line.text` by construction (the new invariant holds without any reconstruction check to retire). A within-ceiling line keeps its whole carrier unchanged (the good short clozes are untouched). The `≤16` eligibility gate (`above_max_token_threshold`) is removed so long lines reach the narrower instead of being dropped; floor+ceiling now apply to the carrier. This subsumes the spec's intent with less risk (no mass-drop, keeps the faithfulness guard) and less churn (sanitizer + prompt rule 1 unchanged). M1 commit-binding satisfied: the `renderContracts.ts:289` stale-comment fix rode the F2 commit.
  - **Flow fix (caught by the L7 canary run):** a narrow-drop (no in-band carrier sentence) is recorded as a **structural SKIP** (`no_viable_carrier_sentence`), NOT a `failedLineRef`. Routing it to failedLineRefs made CS22 ERROR → Stage B `partial` → the publish CLI's `status!=='ok'` exit **aborted before `publishVocabulary`** (the distractor reseed never ran). A narrow-drop is a property of the line (no good cloze exists), like ineligibility — so it's a skip; status stays `ok`, the run completes, and the line's cap still falls out of the emit set → the retire sweep stops any pre-existing over-long cloze rendering.
- **Read-side completeness (W7) — verified safe.** No runtime reader re-asserts "filled blank == original line": `src/components/exercises/feedbackMapping.ts:163,180` only build a *display* string via `sentence.replace('___', target)` (assigned to `promptShown.text`), never compared to the line; the pipeline validator CS11 (`validators/dialogueClozes.ts`) checks blank-count only, no reconstruction. So a contiguous-span carrier renders correctly on the read side. (Grep evidence carried per the load-bearing-claim rule.)
- **Note (R3) — the F1 answer-equality exclusion is belt-and-braces:** the answer's own rendered string is already excluded today (meaning: `answerForms()` + synonym threshold `selectDistractors.ts:145-149`; form: `isExcluded` `:49-53`). The net-new behavior is purely the cross-distractor dedup; the explicit answer-equality restatement is for self-documenting the invariant, not missing coverage.
- **C5 — translation correspondence (data-architect call).** `dialogue_clozes` carries `translation_text/_nl/_en` keyed to the **whole line**; if the carrier is now one sentence, the comprehension aid no longer matches the carrier. Recommended minimum-mechanism resolution: **keep the whole-line translation as the comprehension aid** (it gives the learner *more* context, never less; no new generation). Alternatives (sentence-aligned translation; LLM sentence translation) are more mechanism — default to keeping whole-line unless data-architect objects. This is the writer→reader shape question that makes data-architect a required reviewer.

### F5 — Regeneration path (REQUIRED to fix existing lessons)
A routine re-publish **skips seeded content**: the dialogue per-line `seededLineIds` filter (`generateClozeContexts.ts:360-366`, surfaced via `runner.ts:299-302`) and the distractor seed-once skip (`seedDistractors.ts:105-115`). So F1/F2 only affect *new* lessons unless we force regeneration. Touch points for **`--regenerate-dialogue`** (the union `model.ts:62-64` currently has only `item`/`pattern`; CLI parses only `--regenerate`/`--regenerate-pattern` at `publish-approved-content.ts:46-61`):
1. add a `{ kind: 'dialogue'; … }` discriminant to the regenerate union (`model.ts`);
2. parse a `--regenerate-dialogue` CLI flag (`publish-approved-content.ts`);
3. plumb `regenerate: input.regenerate?.kind === 'dialogue'` into the `generateDialogueClozes` call (`runner.ts:299-302`) — the generator **already accepts `regenerate?: boolean`** (`generateClozeContexts.ts:355,363`) that bypasses the seeded gate;
4. retire + re-upsert the affected `dialogue_clozes` rows for the lesson.
For distractors (W6 — enumerated to the same depth, since F1's L7 acceptance depends on it): the existing per-item `--regenerate <normalized_text>` (`model.ts` `{kind:'item'}`) bypasses the seed-once skip (`seedDistractors.ts:105-115`) for one item. A lesson-scoped `--regenerate-distractors <lesson>` needs: (1) a CLI flag (`publish-approved-content.ts`); (2) a lesson→items expansion (the lesson's item normalized_texts); (3) for each, the existing per-item regenerate bypass at `seedDistractors.ts:105-115`; (4) retire/replace the existing `distractors` rows for those caps before re-seed. Cheaper alternative if F5-distractor is deferred: run F1 on **fresh lessons only** + the F1b one-off cleanup for the 5/43 existing L7 dups — but then the F1 acceptance baseline (below) must be scoped to fresh lessons.
