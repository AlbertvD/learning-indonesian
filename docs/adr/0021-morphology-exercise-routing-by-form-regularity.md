# ADR 0021: Morphology exercise type is routed by affix form-regularity, at projection time, reusing vocabulary cap types

## Status

Accepted (2026-06-22). Design hammered out with the author via a grill-with-docs session. Relates to ADR 0018 (the 2-cap application tier + cross-source-kind prereqs), ADR 0019 (the derivation engine + the reduplication/confix exercise split), ADR 0011 (DB-authoritative-after-seeding / additive re-runs). Supersedes nothing; it *adds a routing fork* to the morphology application tier.

## Context

Every `affixed_form_pair` emits two capabilities (ADR 0018): a recognise cap and a produce cap. Today both render **form** drills regardless of affix â€” `recognise_word_form_link_cap` â†’ `decompose_word_ex` ("segment `berjalan`"), `produce_derived_form_cap` â†’ `type_form_ex` ("type `berjalan`"). For **allomorphic** affixes (`meN-`/`peN-`) that is right: the *form* is the hard, low-salience skill (nasalization). But for a **single invariant prefix/suffix** (`ber-`, `di-`, `se-`, `ter-`, `-an`, `-kan`, `-i`) the form is one trivial move â€” prepend/append a fixed string â€” so the drill is mechanical string-matching, not morphology. The author observed this live (2026-06-22): *"in all cases you simply need to add ber to the base word."* For suffixes it is worse than busywork: `-kan`/`-i` are formally trivial yet semantically the hardest discrimination in the language (research [18][2]), so the form drill trains the easy part and never the hard one.

The research evidence base (`docs/research/2026-06-15-affix-morphology-module-research.md`, SLA update verified 2026-06-22): an affix drill's purpose is to force noticing of the formâ†”meaning link [28]; recognition develops before production [26]; effects are modest (dâ‰ˆ0.33) so do not over-build.

Two facts constrain *where* the fork can live:

1. **The runtime is affix-blind by design.** `validateCapability` (which produces `allowedExercises`) and `resolveExercise` see only `capabilityType` + `sourceKind` â€” never the affix. The affix surfaces only at the reader (`byKind/affixedFormPair.ts`), *after* exercise-type resolution. `capabilityType` is *the* render router (`RENDER_CONTRACTS`).
2. **The substrate already exists but is fragile.** Derived-form meanings (`derived_gloss_nl/_en`, PR #271) and carrier sentences (`carrier_text`) are in the DB, but the glosses are *not* in canonical staging and *not* written by the projector â€” and `replaceAffixedFormPairs` does delete-then-insert, so **any re-publish wipes the glosses to NULL**. Carriers are *harvested* from lesson text, so coverage is partial (61/267) and uneven.

## Decision

**Route the morphology exercise type by the affix's form-regularity, decided once at the capability stage and encoded in the capability type.**

- **Discriminator** (both fields already on the `AFFIX_CATALOG` entry): route to **meaning/usage** iff `affixType âˆˆ {prefix, suffix}` **and** `allomorphClassesFor(affix)` is empty. Else (`meN-`/`peN-` allomorphic, every `confix`, every `reduplication`) **keep formation**. This targets exactly the "one trivial move" class and leaves the deliberate confix/reduplication form split (ADR 0019) intact.
- **Reuse existing capability types â€” mint none.** Transparent pairs emit `recognise_meaning_from_text_cap` (renders the meaning MCQ via `choose_meaning_ex`) and, *only when a carrier exists*, `produce_form_from_context_cap` (renders the usage cloze). Allomorphic/confix/reduplication pairs keep `recognise_word_form_link_cap` + `produce_derived_form_cap`. The produce cap's prerequisite is the pair's own recognise cap; the ADR-0018 root-vocab prereq is unchanged.
- **The vocab/grammar/morphology analytics split keys on `source_kind`**, not cap type (`masteryModel.funnelBucket`: `word_form_pair_src â†’ morphology`; the vocabulary skill profile fences itself to `sourceKind === 'vocabulary_src'`). So a reused *vocabulary* cap type carried on a `word_form_pair_src` pair counts as morphology and never leaks into vocabulary-size counts.
- **Meaning-MCQ distractors are deterministic**, priority-sourced from the morphology data: the root's own bare meaning + same-root family siblings (the richest discriminators â€” they drill the meaning-shift) â†’ same-affix/same-lesson glosses â†’ global same-POS pool. Answer-exclude + dedup + guarantee-3 (a meaning card that cannot reach 3 options would leave the pair with no recognise card). Curated/LLM distractors rejected â€” deterministic selection > LLM generation (project default), modest effects don't justify the cost.
- **Make the glosses republish-safe.** `derived_gloss_nl/_en` become committed authored substrate, threaded through to the projector + insert so a republish reproduces them. The home is a **separate per-lesson `morphology-glosses.ts`** (`source_ref â†’ {nl, en}`), *not* `morphology-roots.ts` â€” the proposer (ADR 0020) rewrites every roots entry wholesale on `--write` (`propose-morphology-roots.ts:283,287`), so a gloss field added there would be clobbered. The glosses file is decoupled from both the proposer and `generate-morphology-patterns` (which merges it in by `source_ref`), so regenerating either preserves it. The existing 267 DB glosses are migrated into it one-time. This is the pipeline-is-writer rule applied to substrate the redesign depends on.
- **Lift carrier coverage deterministically, then fall back to meaning-only.** Expand `harvestCarrier`: take the segment of arrow (`â†’`) grammar examples containing the derived form (currently the whole line is dropped), and add exercise-prompt + dialogue-line tiers (currently only exercise `answer` strings are read). Pairs with no harvestable carrier are meaning-only (no synthetic carriers in v1).

## Considered options

- **Render-time routing (a transparency column on `learning_capabilities`, or `affix` on `ProjectedCapability`)** â€” rejected. It adds a *second* render router beside `capabilityType` and leaks affix-awareness into the affix-blind readiness layer. The omission test kills it: the new signal exists only to recover information the runtime deliberately discards; projection-time routing deletes the need.
- **Mint morphology-specific cap types** (`recognise_derived_meaning_cap`, â€¦) â€” rejected for v1. Strictly more mechanism (enum + CHECK + render contract + mastery branches + guardrails, colliding with the in-flight Â§8 naming rename) for a separation `source_kind` already gives the analytics for free. The reused cap types also describe the skills accurately ("recognise meaning from text"; "produce form in a context").
- **Blunt allomorphy-only signal** (`allomorphClassesFor` empty â†’ meaning) â€” rejected. It sweeps confixes and reduplication into the meaning track, silently reversing ADR 0019's grounded form split. Their form (wrap / doubling) is not the trivial single move the complaint was about.
- **Generate synthetic carriers for all pairs / curated distractors** â€” rejected for v1. Authoring/token cost for a secondary card; harvested real carriers beat synthetic; the deterministic harvest-expansion recovers most of the gap.
- **Keep `type_form_ex` as the transparent produce card when no carrier exists** â€” rejected. Reintroduces the exact trivial "type `berjalan`" card the redesign removes.

## Consequences

- **No schema change.** The cap-type-per-pair decision rides the existing `capability_type` column; `derived_gloss_nl/_en`, `carrier_text`, `affixed_form_pairs` shape, CS12/HC31, and the engine (ADR 0019) are untouched. The work is projector routing + render-contract widening (`choose_meaning_ex` and the cloze exercise gain `word_form_pair_src` support) + reader/builder branches + staging-pipeline plumbing for the glosses.
- **Existing morphology lessons need a re-publish** to swap cap types (build-stage; cheap). The gloss-into-staging migration must land *before/with* that republish or the meaning card goes blank.
- **Dual sign-off required** (CLAUDE.md): touches the capability projection, the writer-reader-validator contract, and the render contract â†’ both `architect` (module placement / seams) and `data-architect` (writer-reader-validator shape) before the implementing plan is `approved`.
- **Planned additive follow-on â€” meaning for *all* affixes.** Give the allomorphic/confix/reduplication track a meaning card *alongside* its form cards, so every affix also practises meaning. Purely additive (one extra `recognise_meaning_from_text_cap` per pair; no rework, no migration), deliberately deferred from v1 (modest effects; the complaint was transparent-affix form busywork) and **gated on a check that the derived form is not already a vocabulary `learning_item`** (high-frequency forms like `memukul`/`makanan` may already carry a vocab meaning card â€” building a parallel morphology one would duplicate it).
- **`choose_form_ex`'s `word_form_pair_src` branch stays dead** (the resolver picks `decompose_word_ex` first for the recognise cap). Out of scope here; noted for a later cleanup.
