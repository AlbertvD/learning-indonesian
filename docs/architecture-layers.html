indonesian.duin.home

Four plates: a browser, a gateway, a database, and the foundry that fills it. The first three are the running app; the fourth is the pipeline that turns coursebook photographs into the contents of the third. Every claim is grounded in a file you can grep.

Plate II · kong & friends

Contract
i.ii

  • Transport. HTTPS to api.supabase.duin.home; Traefik terminates TLS via Step-CA, Kong receives upstream.
  • Identity. Authorization: Bearer <JWT> alongside apikey: <anon>. Kong injects the anon key from its own config.
  • Schema selector. Accept-Profile: indonesian on reads, Content-Profile: indonesian on writes. Set automatically by .schema('indonesian').
  • Session storage. Cookie sb-supabase-auth-token bound to .duin.home; absent on localhost so cookies aren't silently dropped.
  • CORS. Allow-list in kong.yml includes https://indonesian.duin.home and the Plex headers required by supabase-js.
  • Realtime. Not currently used by this app.
II

Plate II · the gateway

The Switchboard

The self-hosted Supabase stack, shared with the family-hub application. Kong fronts everything; PostgREST, GoTrue, Storage and Edge Functions sit politely behind it on the docker proxy network.

Kong — API gateway ii.a

api.supabase.duin.home
TLS via Traefik & Step-CA. Routes /rest, /auth, /storage, /functions. Bakes in the anon key. Enforces the CORS allow-list at services/supabase/kong/kong.yml.

PostgREST — /rest/v1/* ii.b

REST + RPC for the indonesian schema
Exposed via PGRST_DB_SCHEMAS=public,storage,graphql_public,indonesian. Translates client calls — .from('lessons'), .rpc('get_lessons_overview',…) — into SQL, applying RLS using the JWT.

GoTrue — /auth/v1/* ii.c

Email/password auth · JWT issuer
Endpoints in use: /token?grant_type=password, /signup, /logout, /user. GOTRUE_MAILER_AUTOCONFIRM=true; no email flows. Auth state broadcasts to the client via onAuthStateChange.

Storage — /storage/v1/object/public/* ii.d

Public read buckets
Audio served via storage.from('indonesian-lessons').getPublicUrl(…) at lessonService.ts:201; podcasts via storage.from('indonesian-podcasts').getPublicUrl(…) at podcastService.ts:42.
indonesian-lessons indonesian-podcasts

Edge Functions — /functions/v1/* ii.e

commit-capability-answer-report — Deno · ts-fsrs@5.3.2
The server-authoritative FSRS scheduler. Receives an AnswerReport, computes next-due dates, and writes to learner_capability_state and capability_review_events. Sole writer for review outcomes — frontend never schedules.
supabase/functions/commit-capability-answer-report/index.ts

Plate III · postgres

Contract
ii.iii

  • PostgREST → DB. PG wire protocol; JWT claims set as request.jwt.claims; RLS evaluates per-row.
  • GoTrue → DB. Writes the auth schema (users, sessions, refresh tokens); never touches indonesian.
  • Storage → DB. Object metadata in the storage schema; bytes on a disk volume mounted into the container.
  • Edge function → DB. Service-role supabase-js client — bypasses RLS, by design and only here.
  • Schema isolation. The indonesian schema is invisible to other apps on the shared instance; family-hub cannot read it.
  • Migrations. Applied via make migrate; idempotent SQL in scripts/migration.sql and scripts/migrations/*.sql.

Plate IV · the foundry

Contract
iii.iv

  • Schema. make migrate SSHes into the homelab and applies scripts/migration.sql + scripts/migrations/*.sql via docker exec supabase-db; reloads the PostgREST cache. Idempotent.
  • Tables. publish-approved-content.ts <N> writes lessons 4+ via PostgREST; seed-*.ts scripts handle legacy content. Service-role key, never committed.
  • Storage. Audio uploaded with the service role to indonesian-lessons and indonesian-podcasts; public read.
  • Capabilities. materialize-capabilities.ts runs projectCapabilities + validateCapabilities locally and writes the projection to learning_capabilities.
  • External services. Anthropic API for cataloguing & agents; Google Cloud TTS Chirp3-HD for audio; Tesseract for OCR. None of these are runtime dependencies of the app.
  • Health gates. make pre-deploy runs lint + test + build + check-supabase + check-supabase-deep; CI cannot reach the homelab, so this is the manual seam.
IV

Plate IV · the workshop

The Atelier

Where coursebook photographs become published lessons. A deterministic spine — capture, OCR, cataloguing, staging, agent authoring, validation, publication — running entirely from a developer's laptop and pushing finished content into Plate III.

[X] script [A] agent [O] output artifact [E] external service

1. Capture & OCR iv.a

Phone photographs of physical pages, normalised and read by OCR. Local-only; the raw images are gitignored.
convert-heic-to-jpg.ts ocr-pages.ts tesseract content/raw/lesson-N/*.{heic,jpg} content/extracted/lesson-N/page-N.txt

2. LLM Cataloguing iv.b

Claude reads OCR text alongside the photographs, identifies section boundaries, and parses vocabulary, expressions, numbers, dialogue, and grammar into structured items. Each lexical item is POS-tagged for distractor filtering.
catalog-lesson-sections.ts anthropic.com (claude) sections-catalog.json

3. Staging Generation iv.c

Deterministic translation from catalogue → editable TS. Reads sections-catalog.json; writes the lesson scaffolds and a barrel.
generate-staging-files.ts scripts/data/staging/lesson-N/lesson.ts learning-items.ts grammar-patterns.ts candidates.ts cloze-contexts.ts index.ts

4. Linguist Agents iv.d

Claude subagents that author, enrich and review. Run in sequence, ending with the reviewer; review-report.json = approved means publish-ready.
linguist-structurer grammar-exercise-creator vocab-exercise-creator cloze-creator linguist-reviewer pattern-brief.json vocab-enrichments.ts review-report.json

5. Publication iv.e

publish-approved-content.ts <N>  — lessons 4+, the canonical path
Single-shot publish: lesson sections, learning items, grammar patterns, cloze contexts, exercise variants. Quality gates at every step; non-zero exit on failure routes the operator back to the agent that produced the offending file.
publish-approved-content.ts
Legacy seeders — lessons 1–3 + shared infrastructure
Pre-pipeline content lives in scripts/data/lessons.ts, vocabulary.ts, podcasts.ts; these scripts seed it. Same approach handles podcast audio uploads and public flashcard decks.
seed-lessons.ts seed-vocabulary.ts seed-podcasts.ts seed-cloze-contexts.ts seed-learning-items.ts seed-lesson-audio.ts seed-flashcards.ts

6. Capability Materialisation iv.f

After publication, projections of the capability catalogue are computed locally — calling projectCapabilities and validateCapabilities from src/lib/capabilities/ — then written to learning_capabilities with readiness status. The runtime app reads only the result.
materialize-capabilities.ts check-capability-health.ts promote-capabilities.ts auto-fill-capability-artifacts-from-legacy.ts approve-staged-capability-artifacts.ts check-capability-release-readiness.ts

7. Audio Generation iv.g

Synthesised audio for sections, exercises, and spoken variants. Cached in the audio_clips table to avoid re-billing on republish; voices assigned per lesson.
generate-section-audio.ts generate-exercise-audio.ts generate-spoken-variants.ts set-lesson-voices.ts google cloud tts (chirp3-hd)

8. Schema & Migrations iv.h

make migrate SSHes into the homelab and runs SQL via docker exec supabase-db; idempotent. Pre-deploy gate runs lint, tests, build, and the two health checks before merging migration changes.
migrate.ts migrate-run.ts scripts/migration.sql scripts/migrations/*.sql check-supabase.ts check-supabase-deep.ts check-supabase-rls.ts

9. Quality & Repair iv.i

Auditors and repair tools that don't fit a single pipeline stage — coverage checks, translation audits, dialogue-chunk reactivation, page-block sync, end-to-end smoke tests.
lint-staging.ts check-exercise-coverage.ts check-vocab-coverage.ts audit-translation-parens.ts repair-item-meanings.ts repair-stability.ts reactivate-dialogue-chunks.ts regenerate-all-lesson-page-blocks.ts sync-lesson-page-blocks-only.ts test-e2e-pipeline.ts asr-quality-gate.ts

Staging — source of truth iv.j

For lessons 4+, the directory scripts/data/staging/lesson-N/ is the canonical authoring artifact. The DB is a projection of these files; reverse-engineer-staging.ts can reconstruct them from a published lesson when starting from the legacy path.
staging/lesson-N/lesson.ts learning-items.ts grammar-patterns.ts candidates.ts cloze-contexts.ts vocab-enrichments.ts sections-catalog.json pattern-brief.json review-report.json data/lessons.ts (legacy 1–3)

The boundary with src/lib/ iv.k

The pipeline imports projectCapabilities, validateCapability, and buildCanonicalKey directly from src/lib/capabilities/. The runtime app calls validateCapability exactly once (defensive re-check at session-load); everything else under capabilityCatalog, canonicalKey, and capabilityContracts fires only here. The shared module is the contract that keeps pipeline-time keying and runtime-time keying in lock-step.
